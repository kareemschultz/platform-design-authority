Add `scripts/generate_design_tokens_css.py` (RR-004 / TD-004, issue #21): the `--status-*`, `--border-strong`, and `--border-strong-overlay` CSS custom properties in `globals.css`'s `:root`/`.dark` blocks are now generated from `registry/design-tokens.json` between `/* BEGIN/END generated design tokens */` markers, with `--check` drift detection wired into `bun run gates` and Documentation Governance CI, instead of being hand-copied literals a human had to remember to keep in sync. Contrast-rationale comments are rendered from each token's registry `$description` field, which is now the single source for both the value and the rationale (`tokens.color.dark.border-strong` gained the one missing `$description` this required). No consumer-visible color or contrast change: every generated hex value and every previously-verified contrast ratio (border-strong 4.76:1 light / 3.76:1 dark, border-strong-overlay 3.58:1 light / ~4.25:1 dark, status-warning-foreground 7.09:1 light / 10.73:1 dark) is unchanged, verified by a pre/post numeric-token diff. The non-status semantic aliases (`--action-primary`, `--surface-*`, `--text-*`, `--focus-ring`) are untouched and remain the recorded fifth-audit F-H-003 controlled-prototype exception.
