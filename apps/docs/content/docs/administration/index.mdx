# Administer the local prototype

This guide is for developers or reviewers using a disposable local WS1 prototype. The application now provides a thin Administration shell for current identity, context, users, roles, entitlements, sessions, and audit evidence. It is not a production administration console.

## Authority boundary

Access to Docker or the local machine is infrastructure access, not an application permission. Better Auth owns authentication and sessions, not Party identity, business roles, permissions, entitlements, or tenant hierarchy. Every protected operation is checked by the server even when its link is visible.

## Use the Administration shell

1. Open the web application and choose **Sign in**.
2. After authentication, open **Administration**.
3. The application establishes a server-validated active organization context for an active membership. The context is scoped to the browser tab.
4. Use the visible organization and location selectors to change scope. A context change does not grant new authority.
5. Use **Users**, **Roles**, **Entitlements**, **Sessions**, and **Audit** from the second-level navigation.

Users and memberships, roles and permissions, and entitlements are separate concepts. A provisioned entitlement does not grant a person permission, and a permitted person cannot use an unprovisioned capability.

The shell distinguishes **Permission denied**, **Capability unavailable**, **Additional verification required**, **Approval required**, **Sign in again**, and **Offline**. Do not treat those states as interchangeable.

## Session safety

The Sessions page lists the current user's active sessions. Revoking another session causes that device to fail its next protected request. Revoking the current session signs this browser out. Device labels and masked network evidence are informational and must not be used as authentication factors.

## Current limitations

- Factor enrollment, recovery, and production OTP delivery are not available in this shell.
- Invitations, membership suspension, and role assignment are implemented as governed API capabilities but do not yet have shell forms.
- Audit metadata is not expanded in the shell because it may contain classified evidence.
- Offline mode does not authorize changes and does not replace current server authority.
- Production support access, impersonation, secret operations, backup, failover, and incident controls remain blocked.

## Required configuration

The server validates these variables at startup:

| Variable | Requirement |
|---|---|
| `BETTER_AUTH_SECRET` | At least 32 random characters |
| `BETTER_AUTH_TRUSTED_ORIGINS` | Comma-separated exact origins |
| `BETTER_AUTH_URL` | Server URL, normally `http://localhost:3000` |
| `CORS_ORIGIN` | Exact web origin, normally `http://localhost:3001` |
| `DATABASE_URL` | PostgreSQL connection string |
| `NODE_ENV` | `development`, `production`, or `test` |
| `NEXT_PUBLIC_SERVER_URL` | Browser-visible server URL |
| `PLATFORM_API_INTERNAL_URL` | Optional server-only API origin for container or private-network rendering; Compose uses `http://server:3000` |

Do not add paths, queries, fragments, or trailing slashes to trusted origins.

## Start and inspect services

```bash
bun run docker:up
docker compose exec server bun run db:migrate
bun run docker:logs
```

The PostgreSQL service must report healthy before schema or authentication operations. The database is intentionally not published to the host by Compose; application containers reach it through the Compose network. Direct source-mode server development therefore requires a separately configured host-reachable PostgreSQL 18 instance.

## Health checks

| Check | Command | Expected result |
|---|---|---|
| API root | `curl --fail http://localhost:3000/` | `OK` |
| Web root | `curl --fail http://localhost:3001/` | Successful HTTP response |
| PostgreSQL | `docker compose exec -T postgres pg_isready -U postgres` | Accepting connections |
| Version | `docker compose exec -T postgres psql -U postgres -d meridian -Atc "SHOW server_version;"` | `18.4...` |
| Extensions | `docker compose exec -T postgres psql -U postgres -d meridian -Atc "SELECT extname FROM pg_extension ORDER BY extname;"` | `pg_stat_statements` and `plpgsql` only |

ADR-0024 prohibits adding optional extensions merely for convenience.

## Backup and restore

For synthetic local data, create a custom-format dump:

```bash
docker compose exec -T postgres pg_dump -U postgres -d meridian --format=custom > meridian.dump
```

Restore only into a disposable, compatible local environment:

```bash
docker compose exec -T postgres pg_restore -U postgres -d meridian --clean --if-exists < meridian.dump
```

These commands are prototype notes, not production backup or disaster-recovery evidence. Production requires encrypted storage, access control, retention, monitoring, point-in-time recovery, restore drills, privacy-journal replay, and documented recovery objectives.

## Reset local data

```bash
docker compose down -v --remove-orphans
bun run docker:up
docker compose exec server bun run db:migrate
```

This permanently removes the local Compose volume. Confirm that the environment contains only disposable synthetic data before running it.

## Stop services

```bash
bun run docker:down
```

Use `docker compose down -v --remove-orphans` only when a complete disposable reset is intended.

These limits remain governed by the Platform Constitution, PDA-RDM-008, ADR-0005, ADR-0020, and the first-slice specifications. Implementation evidence is recorded in PDA-UX-038.
