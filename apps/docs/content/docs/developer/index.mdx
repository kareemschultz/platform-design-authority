# Develop the platform prototype

This implementation is controlled prototype evidence inside the Platform Design Authority repository. Root `AGENTS.md`, `CLAUDE.md`, the Constitution, accepted decisions, governed specifications, registries, and contracts apply before nested implementation guidance.

## Repository layout

```text
apps/
    docs/       Fumadocs documentation portal on port 4000
    native/     Expo/React Native prototype
    server/     Hono server with Bun and Node entries
    web/        Next.js web prototype on port 3001
packages/
    contracts/          generated and hand-owned API, capability, event, and permission contracts
    domains/            runtime-neutral Catalog, Inventory, and Party behavior
    foundation/         money, quantity, time, identifier, and result primitives
    persistence/        owner-specific PostgreSQL adapters, schemas, and migration histories
    platform/           identity, tenancy, authorization, entitlements, audit, events, and numbering
    platform-clients/   typed client adapters
    tooling/            shared configuration and validated environment schemas
    ui-web/             platform-owned shadcn-derived web source
ops/postgres/ local PostgreSQL prototype controls
```

Architecture authority lives in `docs/blueprint/`. Implementation notes live in `docs/implementation/`; canonical API and message contracts remain in root `openapi/` and `schemas/`.

## Install and verify

```bash
bun install --frozen-lockfile
bun run check
bun run check-types
bun run test
bun run build
bun audit
```

The CI workflow additionally verifies the Node fallback, registry freshness, prohibited surfaces, Compose configuration, PostgreSQL 18.4 baseline, and Docker images.

## Run one application

```bash
bun run dev:web
bun run dev:server
bun run dev:native
bun run --filter docs dev
```

Use `bun run dev` to start the Turbo development graph. Environment variables are validated by `packages/tooling/env`; do not bypass validation except in a narrowly controlled build step that cannot access runtime data.

## Add an API procedure

Before implementation:

1. Identify the owning platform area, engine, or domain.
2. Confirm the capability, event, and permission identifiers in root registries.
3. Confirm first-slice depth and lifecycle authority.
4. Define tenant, actor, permission, entitlement, idempotency, audit, privacy, offline, and failure behavior.
5. Use an application contract; never import another domain's private persistence.

Application contracts belong in `packages/contracts/platform-api`; runtime-neutral behavior belongs to its owning Platform or Domain package. `apps/server` is the composition root that binds persistence, application implementations, request context, and oRPC/Hono transport. Provider SDKs and UI components must not become business abstractions.

## Add database state

The owning domain controls its schema and migrations. Do not create cross-domain foreign persistence shortcuts. Use PostgreSQL-native features first and consult ADR-0024 before adding any extension.

Generate and review migrations:

```bash
bun run db:generate
```

Apply reviewed migrations to a disposable local database:

```bash
bun run db:migrate
```

See [Migrations](/docs/migrations) for rollback and evidence requirements.

## Add UI source

Shared web primitives live in `packages/ui-web/core`. New shadcn source must use the pinned CLI, preserve Base UI/Rhea configuration, record provenance, and be normalized to platform tokens and state conventions.

```bash
bunx shadcn@4.13.0 add dialog popover -c packages/ui-web/core
```

Review generated code before keeping it. Cover loading, empty, error, denied, offline, conflict, and success states plus keyboard, focus, screen-reader, zoom, contrast, touch, responsive, motion, and white-label behavior.

## Update documentation

Update product documentation in the same change when behavior, configuration, API, permissions, migrations, operations, or troubleshooting changes. Product prose describes released or explicitly labelled prototype behavior only. Architecture decisions stay in governed root documents.

Run from the repository root:

```bash
python scripts/validate_docs.py
python scripts/generate_registries.py --check
python -m unittest scripts/test_validate_product_docs.py
python scripts/validate_product_docs.py
bun run --filter docs types:check
bun run --filter docs build
```

## Runtime compatibility

Bun is the preferred prototype runtime. The server must keep a working Node fallback. Isolate runtime-specific APIs behind adapters, retain standards-based Web APIs where practical, and record compatibility findings in PDA-ENGR-013 before changing pins or workarounds.

## Production boundary

Passing prototype CI does not authorize production. Production still requires accepted governing decisions, implementation evidence, tenant-isolation tests, security and accessibility review, backup/restore exercises, provider certification where applicable, operational readiness, and release approval.
