# Controlled platform prototype — evidence refresh 2026-07-16

This is an internal implementation-evidence checkpoint for product prototype `0.0.4`, evaluated at `main` commit `7202fc819b70982c013e1ca11a4fcc136e01e2de`. It is not a customer release, production-readiness declaration, compatibility promise, or support window.

## Included at the evidence revision

- A Bun/Turborepo modular-monolith workspace with runtime-neutral foundation, contract, Platform, Domain, and owner-specific persistence packages.
- Hono and oRPC composition with a governed Node fallback, typed errors, active-context enforcement, permission checks, idempotency, and version preconditions where declared.
- Better Auth for authentication and sessions, kept separate from Party, tenant membership, business roles, permissions, and entitlements.
- WS1 Tenancy, Party linkage, authorization, entitlements, Audit, session revocation, and a responsive Administration shell with real read/query behavior.
- WS2 Catalog Product/Variant/Identifier behavior and Inventory adjustment, count, transfer, reservation, balance, and linked-reversal ledger behavior through merged PR #69.
- The merged PR #74 Event Backbone controlled prototype: transactional delivery, bounded retry/dead-letter behavior, registered Catalog search and Inventory reconciliation consumers, permissioned bounded replay, and `OPS-SVC-005` procedure-draft operations guidance.
- Deterministic owner-specific PostgreSQL migration streams for Platform, Party, Catalog, and Inventory packages.
- A PostgreSQL 18.4 Compose baseline with `pg_stat_statements` as the only preloaded optional extension.
- A repository-owned Fumadocs portal with local search and LLM-readable routes.
- CI gates for frozen install, formatting, typecheck, tests, builds, Node fallback health, architecture rules, registry freshness, dependency/security checks, PostgreSQL migration/smoke evidence, and Docker images.

## Documentation evidence controls

- Every MDX page has a stable `PDOC-NNNN` identifier, audience, content class, publication state, applicable version, evidence revision, and implementation/contract references.
- `registry/product-documentation.json` is the page inventory and evidence manifest; it remains separate from architecture authority in `registry/documents.json`.
- The product-doc validator rejects missing pages, duplicate IDs/routes, metadata drift, stale or non-ancestor evidence revisions, unknown capabilities/permissions, missing evidence/contracts, broken internal routes, and false generated-API parity.
- The API page separates the implemented runtime boundary from a generated canonical 100-operation Draft reference; generator freshness and exact OpenAPI operation parity are enforced.

- POS, register, cash, stored value, payment-provider, tax, fiscalization, Finance, offline transaction, or reporting user workflows.
- Production native authentication exchange, email/OTP provider delivery, broad recovery, or customer-support operation.
- Production RLS topology, Event Backbone capacity/SLOs/dashboards/alerts, multi-replica evidence, secrets/deployment approval, backup/restore exercises, observability and incident evidence, penetration testing, WCAG conformance evidence, or provider certification.
- AI chat, interactive API proxy, payment facilitation, custody, marketplace code execution, or production business automation.
- Founder, customer, Guyana professional, legal, regulatory, banking, payment, or commercial decisions still named by the blueprint.

## Compatibility notes

- Bun 1.3.14 is the preferred prototype runtime; the Hono server retains a Node 24 fallback.
- Dependency pins and available upstream versions are distinguished in the technology lifecycle ledger; an available patch is not automatically adopted.
- Deep Windows checkout paths can exceed build-tool path limits; use a shorter local path or the evaluated Linux CI result.
- PostgreSQL extensions remain governed by ADR-0024 and are absent unless explicitly allowed.

## Upgrade and support status

No compatibility or support window is promised for this prototype checkpoint. Dependency changes require current technology evidence, a reviewed diff, frozen-lock validation, runtime/build tests, and updated lessons. Customer-facing release notes begin only after an approved product release and release record exist.
