# Use the current web prototype

The current prototype demonstrates connectivity, email-and-password authentication, current tenant context, Party linkage, role and entitlement inspection, session revocation, audit inspection, responsive administrative navigation, and server-side permission enforcement. It is not a released retail application. Catalog and Inventory web workflows are not present at this evidence revision.

## Available actions

| Action | Authentication | Current behavior |
|---|---|---|
| View the home page | Not required | Shows whether the API health query connected |
| Create a prototype account | Not required | Creates a Better Auth user and session |
| Sign in | Not required | Starts a session for an existing account |
| Open the dashboard | Required | Redirects unauthenticated visitors to `/login` |
| Open Administration | Required plus active membership | Establishes a server-validated organization/location context |
| Inspect Party identity | `party.record.read` | Shows the Party linked to the active membership when available |
| List tenant users | `platform.user.read` | Shows authentication and membership summaries without merging those concepts |
| List roles | `platform.role.read` | Shows tenant roles and the count of assigned permissions |
| List entitlements | `platform.entitlement.read` | Shows provisioned capabilities separately from actor permission |
| List audit evidence | `platform.audit.read` | Shows tenant-scoped append-only evidence without expanding classified metadata |
| Review or revoke own sessions | Required | Lists the current account's sessions and supports idempotent revocation |
| Change color mode | Not required | Changes the local light/dark/system presentation |
| Sign out | Required | Ends the session and returns home |

Account creation does not assign a tenant, organization, Party, domain role, permission, or entitlement. Prototype provisioning is a separate controlled setup step.

## Create an account

1. Select **Sign In** in the header.
2. If the sign-in form is visible, select **Need an account? Sign Up**.
3. Enter a name with at least two characters.
4. Enter a syntactically valid email address. Use synthetic data in prototype environments.
5. Enter a password with at least eight characters.
6. Select **Sign Up**.

On success, the application opens the dashboard and displays a success notification. On failure, the form displays the Better Auth error without creating a business identity.

## Sign in

1. Open `/login`.
2. Select **Already have an account? Sign In** if necessary.
3. Enter the account email and password.
4. Select **Sign In**.

The application opens `/dashboard` after a successful sign-in. A failed attempt remains on the form and displays an error.

## Use the dashboard and Administration

The dashboard requires a current session. The Administration shell additionally requires an active membership and establishes a server-validated context. Use the organization and location selectors only to choose among authorized scopes; changing context never broadens authority.

Administration separates authentication identity, Party, membership, role, permission, and entitlement. It distinguishes permission denial, unavailable entitlement, additional verification, approval, reauthentication, and offline failure. Invitations, membership suspension, and role assignment have governed API seams but no shell forms at this revision.

The web application does not yet provide sales, Catalog, Inventory, register, customer, reporting, offline transaction, or financial workflows. Those remain governed requirements or API-only prototype behavior, not released user tasks.

## Sign out

1. Open the account menu showing your name.
2. Confirm the displayed email belongs to the prototype account.
3. Select **Sign Out**.

The application ends the session and returns to the home page.

## Accessibility and responsive use

The forms use labels, native input types, inline validation text, disabled submission states, and keyboard-operable owned UI primitives. Formal WCAG 2.2 AA evidence is not yet claimed. Report keyboard, focus, zoom, contrast, screen-reader, touch-target, or viewport problems as prototype defects.

## Offline and degraded behavior

- Authentication and the protected dashboard require the server and database.
- The home page reports a disconnected API when the health query fails.
- The PWA seam does not authorize offline business transactions or administration mutations.
- Do not treat cached presentation as current authority.

## Data and support boundaries

Use synthetic data only. This prototype has no customer-support workflow, account recovery, production email/OTP delivery, privacy request UI, or production retention policy. Tenant administration is inspection-oriented and incomplete. Reset disposable local data only through the operator procedure in [Administration](/docs/administration).
