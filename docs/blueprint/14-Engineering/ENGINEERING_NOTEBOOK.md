# Engineering Notebook

An append-only engineering notebook: patterns, mistakes, discoveries, trade-offs, benchmarks, performance experiments, and migration notes that are **not load-bearing enough for an ADR and not a technology-version fact for the lifecycle ledger, but too valuable to lose**.

## Rules

1. Append-only; entries are never rewritten. Corrections are new entries referencing the old one.
2. Every entry: date, author (human or agent), context, observation, and — where applicable — evidence paths.
3. Nothing here is authority. When a note becomes load-bearing, it graduates: technology/version facts to `TECHNOLOGY_LIFECYCLE_AND_LESSONS.md`, decisions to an ADR, defects/risks to the Architecture Risk Register. The graduating PR records the promotion here.
4. Notes must not restate specifications; link instead.

## Entries

### 2026-07-12 — Notebook established
Created as part of the program-plan batch (PDA-RDM-007). Seed observations from the audit/remediation cycles worth keeping visible:

- **Windows contributor friction:** deep checkout paths break Turbopack builds (MAX_PATH); `core.autocrlf` without `.gitattributes` produced 93 phantom lint errors until LF enforcement landed. Linux CI is authoritative; local Windows failures should be reproduced in CI before debugging locally.
- **tsc incremental caches trip CI content scans:** `tsc --noEmit` with Next's `incremental: true` writes `tsconfig.tsbuildinfo` whose `node_modules` path lists false-positive naive greps. Scans must exclude build artifacts by class, not by name.
- **`import.meta.main` dies under tsdown bundling** (module moves into a shared chunk); gate Bun-serve entry points on `"bun" in process.versions` instead.
- **Registry-first governance works:** the generated-registry + freshness-check pattern (documents, capabilities, events, permissions, endpoint-permissions, first-slice, tests) caught real drift repeatedly across four audits; extend the same pattern to any new machine-readable artifact rather than inventing bespoke checks.
- **Line-length between prose and machine artifacts is where drift lives:** every recurring audit-finding class was a prose claim without a machine mirror (chart tokens, endpoint tables, environment names). Default to giving normative claims a lintable home.

### 2026-07-16 — Multi-agent independent review discipline
Observed and repeated across the WS1 closeout and WS2 PR1-PR4 review cycles (issue #56/PR #57 RR-011; PR #65, #67, #69, #74; issue #70). Recorded here so any agent picking up this repo — a fresh session, a different host, a different model — can reproduce the same discipline instead of re-deriving it. Not a mandate; graduates to an ADR or the risk register if it ever becomes load-bearing rather than a working habit.

Pattern, when one agent (author) hands a governance document, a documentation-only remediation, or a real implementation PR to another agent (reviewer) for independent audit:

- **Isolated checkout, not the working tree in use for other tasks.** Review in a fresh clone or worktree so the audit can't be contaminated by uncommitted state from unrelated work, and so a `git diff`/`git log` against remote refs is unambiguous.
- **Diff against the exact previously-reviewed commit SHA**, not "the PR" as a moving target. Record that SHA in the review comment. A second review round diffs against the first round's reviewed head, not against the PR's base branch — this is what makes "re-review only what changed" possible instead of re-litigating already-cleared work.
- **Fan out parallel verification agents across independent lenses** for large diffs (correctness, architecture-rule/registry compliance, test-dimension coverage, security/tenant-isolation) rather than one linear read. Each lens reports findings independently; merge and de-duplicate before writing the review.
- **Personally re-verify the single highest-stakes claim directly** — actually run the migration, actually hit the endpoint, actually diff the generated registry — rather than trusting a delegated agent's or the author's narrative summary of it.
- **Run live gates, not narrative.** `bun run check-types`, `bun test`, `bun run check`, `python scripts/validate_docs.py`, `python scripts/generate_registries.py --check` (and any workstream-specific validator, e.g. `validate_program_status.py`) get executed by the reviewer, not accepted on the author's say-so that "all green."
- **Post evidence, not a vibe.** Either concrete inline findings tied to file:line, or explicit concurrence that names the exact head SHA reviewed and what was checked. Never a bare "LGTM."
- **Comment via a JSON-payload file, not inline shell quoting.** On Windows/Git-Bash, `gh api ... -f body="..."` silently drops content on unescaped backticks, and `VAR=value cmd` leaks into argv. Build the comment body as a file and pass it with `--input`/`-F body=@file`.
- **Pair every doc patch with an issue/RR-###/TD-### reference.** A silent edit to a governance document, even a correct one, loses the "why" that later review depends on.
- **After every push, check for `chatgpt-codex-connector` bot review comments and act on legitimate findings without being asked**, then reply and resolve the thread.
- **The reviewer never merges.** Findings and concurrence are posted for the human or the PR owner to act on.

### 2026-07-18 — A DB uniqueness constraint written for one use case silently blocked a later, legitimate one (WS3 PR3)

While implementing WS3 PR3 (Return/Refund/Void/Reissue/Exchange, `claude/ws3-integration`), the live-PG lane caught a real bug the domain-level unit tests (in-memory fakes) could never have caught: `inventory_stock_movement_tenant_reversal_uidx`, a partial unique index WS2 added so `reverseAdjustment` could cap an Adjustment's posted movement at exactly one `Reversal` ever, also silently capped a Sale's posted movement at one reversal — but WS3's Return flow legitimately needs MULTIPLE, separate compensating `Reversal` movements against the SAME original Sale movement over time (one per partial Return on the same sale line). The second of two sequential, both-legitimate partial-return approvals would have hit a raw Postgres unique-violation instead of a clean domain error.

Fix: rescoped the index's `WHERE` predicate to exclude `source_type = 'Sale'` rows (`packages/persistence/inventory-postgres/src/schema/inventory.ts`), preserving the original Adjustment invariant exactly while permitting Sale-sourced reversals to repeat against the same original movement — the actual over-return cap lives in `@meridian/domain-pos`'s own Sale-row-locked `sumReturnedQuantity` check, not in this index. A dedicated live-PG regression test (`apps/server/composition/returns.integration.test.ts`, "posts two SEPARATE compensating movements for two sequential partial returns against the SAME original sale movement") proves it.

Lesson: an in-memory repository fake can prove application-layer logic but cannot prove a real database CHECK/unique-index invariant one bit — and a constraint one stage adds for its own narrow use case can quietly become a correctness bug for a LATER stage's different, equally legitimate use case of the SAME shared table. Before reusing another domain's existing table/column for a new purpose, read its constraints for what invariant they actually encode (not just what they're named), and re-run the live-PG lane, not just the unit suite, before declaring a cross-domain extension done.

### 2026-07-20 — Whole-repo governance audit: stale-state and enforcement-drift lessons
Author: Claude (Fable 5 planning session). Context: the audit behind `docs/project/GOVERNANCE_REMEDIATION_PROGRAM_PLAN.md`. Observations worth keeping visible:

- **Audit from `origin/main`, never from the session's checkout.** A three-agent audit initially "found" CLAUDE/AGENTS drift, stale PROGRAM_STATUS rows, and an unclosed risk row — all artifacts of auditing a dead branch that had forked a week earlier. Every finding was re-verified against `origin/main` before planning; roughly a third evaporated. First step of any repo audit: `git fetch` and diff the checkout against `origin/main`; treat a branch whose upstream reads `[gone]` (`git for-each-ref --format='%(upstream:track)'`) as dead until proven otherwise.
- **Unmerged research effectively does not exist.** The entire competitive-intelligence corpus (~100 files, PR #75) sat unmerged for four days, and the founder's working belief became "that research was never done." Research and audit output must merge promptly — its value decays to zero while stranded, and later sessions re-plan work that already exists.
- **Worktree sprawl has a concrete failure mode:** a scratch worktree holding `main` checked out blocks `git checkout main` in the primary tree, and `git worktree remove` refuses on untracked leftovers (`node_modules`). Workaround: branch new work directly off `origin/main` (`git checkout -b <branch> origin/main`) without touching local `main`; keep agent scratch-worktree directories gitignored so they never look like repo content.
- **Path-triggered freshness checks cannot catch externally-induced staleness.** A status validator that only runs when the status file changes goes blind when the world changes instead (an issue closes, a board is created). Freshness gates need a `schedule:` trigger and an explicit last-updated age bound, and scheduled failures need to open an issue — a red run on `main` that nobody watches is not a gate.
- **Prose that claims enforcement is a defect class of its own.** "Raw palette values are lint-visible defects" was written while no lint rule existed; "agents" named as reviewers were actually optional Skills that no gate requires. Extends the 2026-07-12 registry-first lesson: when a normative sentence says "is enforced/lint-visible/required," CI must be able to point at the enforcing step, or the sentence must say "should be."
- **The `docs/project` link validator treats backticked path-like strings as internal references.** Naming a planned-but-not-yet-created document with a backticked path ending in ".md" fails `validate_docs.py` (this very entry originally did so); name future documents in prose (directory in backticks, filename bare) until they exist.

### 2026-07-20 — Implementing the remediation program: git, secrets, and Studio MCP/CLI lessons
Author: Claude (Sonnet 5 implementation session). Context: executing `docs/project/GOVERNANCE_REMEDIATION_PROGRAM_PLAN.md`'s PR-C through PR-9 in one long session, ~20 PRs. Every item below either caused a real mistake this session or was caught before it became one — recorded so the next agent doesn't have to rediscover either.

**Git operations that silently destroy uncommitted work:**

- **`git checkout <ref> -- .` (or `-- <file>`) overwrites the working tree from `<ref>` with no warning and no diff shown first**, even mid-task with real uncommitted edits. Happened twice this session: once restoring a stale local `main` on top of active work, once undoing an intentional edit by using `HEAD` when the actual intent was "discard this one accidental change" (the safe form for that is `git diff` first, then a scoped revert of only the unwanted hunk, not a blanket ref-checkout). Before running any checkout-from-ref command, run `git status` and `git diff --stat` and confirm nothing precious would be silently replaced.
- **A local `main` branch left unpulled for the session's duration silently diverges from `origin/main`**, and `git checkout main -- .` or similar then restores *stale* content over current work, reintroducing already-fixed defects (old plan checkboxes, old CI wiring, old script versions all came back at once this way). If local `main` is ever consulted, `git fetch && git log main..origin/main --oneline` first, or better, never consult a local branch ref at all — use `origin/main` explicitly for every comparison and every `checkout -b <new-branch> origin/main`.
- **After a squash merge, the local branch that produced the PR is *not* an ancestor of the new `origin/main` tip** — its own committed content is gone from `origin/main`'s history graph even though it's fully absorbed by value. Never keep building the next change on a branch whose PR just merged; always `git stash` (if anything is in progress) → `git fetch origin` → `git checkout -b <new-branch> origin/main` → `git stash pop`, for every single PR in a sequence. Skipping this produced stale evidence-cutoff SHAs and once caused a checked-off plan item to silently revert.
- **The Bash tool's working directory does not reliably reset between calls** — it persisted across at least one `cd` this session (contradicting an earlier assumption stated in a prior session transcript), then reset unexpectedly later. Never assume cwd from memory; run `pwd` (and `git branch --show-current` when a git command is about to run) immediately before any operation whose correctness depends on location.
- **An ancestor check (`git merge-base --is-ancestor <branch> <base>`) alone cannot detect a squash or rebase merge** — the merged commit's lineage does not include the source branch's commits, so a genuinely, completely merged branch reads as "not merged." A stale-branch reporter must check the pull request's own `mergedAt` record first, falling back to ancestry only for merges that didn't go through a PR (`scripts/report_stale_branches.py`'s `classify()`). Live-validated: fixing this order found 19 already-merged branches an ancestor-only check had missed, several from earlier in the same session.
- **`gh pr merge --auto --delete-branch` does not reliably delete the branch** when the actual merge is deferred to GitHub's own auto-merge queue rather than performed by `gh` itself at call time — there is no client present at merge time to issue the delete. Confirm with `gh ls-remote --heads origin <branch>` or a scheduled stale-branch report; don't assume `--delete-branch` succeeded just because the flag was passed.

**Encoding and pattern-matching gaps that produced real (if minor) security-relevant misses:**

- **`subprocess.run(..., text=True)` without an explicit `encoding="utf-8"` decodes a subprocess's UTF-8 stdout using the Windows console codepage**, silently mangling any non-ASCII byte (an em dash, here) into mojibake and producing false drift in anything that compares the result to expected text (`scripts/sync_labels.py`'s `gh label list` call). Always pass `encoding="utf-8"` explicitly on Windows; never rely on the platform default.
- **The same class of bug runs the other direction**: a Python script's own `print()` of a Unicode character (again, an em dash) can be mangled when stdout is redirected to a file instead of a live console, because the redirected stream falls back to a different default encoding. The robust fix isn't fighting console encoding — it's not putting non-ASCII separator characters in machine-facing script output in the first place (`scripts/report_stale_branches.py` uses `" -- "`, not `" — "`).
- **A secret-scanning regex written against bash-style `KEY=value` or `KEY: value` syntax misses JSON's `"KEY": "value"`** — the closing quote character sits where the pattern expects literal whitespace before the colon. Confirmed this gap in an already-merged CI secret scan by constructing a real probe (`{"license_key": "<realistic-looking-value>"}`) and observing zero match; fixed by allowing an optional quote before the delimiter.
- **Making a secret pattern also match a legitimate `${VAR}` placeholder is a real risk once the pattern is fixed to be less strict** — widening a regex to catch JSON-quoted secrets can start flagging the exact safe `"license_key": "${LICENSE_KEY}"` config pattern this session introduced for the Studio CLI registries. A negative lookahead (`(?!\$\{)`) is required, and `grep -P` (PCRE) needs `LC_ALL=C.UTF-8` set explicitly to work reliably outside a full UTF-8 locale — verify a corrected pattern against three cases every time: the safe placeholder (must not match), a real-looking secret value (must match), and the live repository (must be silent).

**A new markdown-artifact class needs an explicit accounting entry, every time:**

- `scripts/validate_document_indexes.py`'s `AUXILIARY_DOCUMENT_ROOTS` allowlist is not automatically extended when a new kind of auxiliary file is introduced — adding four slash-command markdown files under `.claude/commands/` failed artifact accounting until the constant was extended to include that directory. Any future new markdown-bearing directory (a new tool's config files, a new automation's templates) needs the same explicit addition, or `validate_document_indexes.py` fails on the very first file placed there.
- Relatedly: `python scripts/generate_registries.py --check` fails after adding a new governed document (front matter with a `document_id`) until `python scripts/generate_registries.py` (no `--check`) is run once to actually regenerate `registry/documents.json`. `--check` only *verifies* the registry is current; it never updates it.

**Credential handling — a bright line, not a judgment call:**

- **Never write a real API key, license key, or account email into any file, gitignored or not, tracked or not, even when the user pastes the literal value into chat and explicitly asks for it to be saved.** The prohibition is about the *action* (an agent entering credential material into any field), not about where it ends up or who asked. The correct response every time: state the exact file path and exact content the user should paste themselves, and decline to be the one who writes it — even to a file the user just created for that purpose.
- A `${VAR}`-style placeholder referencing a credential's *name* (not its value) in a tracked config file is a completely different, safe action — `components.json`'s Studio registries block is exactly this pattern, and it's fine to author directly.
- Moving or copying an *existing*, already-user-created credential file (`.env`) between directories via a plain filesystem operation (`mv`/`cp`) without ever reading its contents is also safe — it's mechanically identical to relocating a sealed envelope. This was necessary because a CLI tool's env loader reads from the current working directory it's invoked from, not the monorepo root, so a monorepo with multiple `components.json` files (one per app/package) needs the credential file physically present next to *each* one.
- `.env` is gitignored by design, so it **never propagates via git** — merging a `components.json` change to `main` does not give any other worktree, clone, or the primary checkout a working credential file. Each physical location needs its own manually-created copy. This surprised both the agent and the user this session.

**A third-party CLI tool can write outside the intended repository entirely when run from a git worktree:**

- Running `npx shadcn add` for a `ui`-aliased single-file component (resolved through a package alias, not the `components` alias) from inside this repository's git worktree (nested under `.claude/worktrees/<name>/`) produced a stray file at the *primary checkout's root* — not inside the worktree at all — twice in direct succession, with two different components (`button.tsx`, then `badge.tsx`). The workspace symlink resolution itself was verified correct (`node_modules/@meridian/ui-web` genuinely points inside the worktree); the CLI appears to compute its write target using a relative-path assumption that doesn't account for a worktree's extra directory nesting versus a normal checkout. This is not fixable from repository configuration — it's documented as a known constraint in `docs/blueprint/09-UX/COMPONENT_INTAKE_FAST_PATH.md` with a concrete mitigation: prefer the MCP content-fetch tools over the CLI when working from a worktree (the agent places the file itself, so there is no third-party write-target computation to escape), and if the CLI route is used from a worktree anyway, check `git status` at the *primary checkout's root* immediately afterward, not only inside the worktree.
- The MCP server itself (`shadcn-studio-mcp`) is registered at the Windows-account (`--scope user`) level, not per-repository — its tools are available in every worktree and every repository in a session regardless of any `.env` file. Only the CLI's authenticated HTTP registry route depends on a local, per-directory `.env`. Don't conflate the two when diagnosing "is Studio available here."

**Re-verify plan assumptions against live state before executing them, every time:**

- Several remediation-plan items turned out to already be resolved, or based on a premise that didn't hold once checked directly: a "fix agents-vs-skills terminology" item had no live instances of the defect to fix; an "close issue #59" item was only 5-of-8 satisfiable by any agent, the other 3 being confirmed founder-only manual actions with no API path; a "refresh PROGRAM_STATUS.md" item found the document was already accurate against all 15 cited issue numbers. In every case, the honest and useful move was to record the corrected finding in the plan next to the original item, not to silently do (or silently skip) what the plan literally said. A plan is a starting hypothesis about the repository's state, not a substitute for checking it.
