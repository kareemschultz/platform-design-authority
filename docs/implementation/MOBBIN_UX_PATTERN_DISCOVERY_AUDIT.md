# Mobbin UX Pattern Discovery Audit

## 1. Executive Summary

Mobbin is a real-world UX pattern reference tool — a searchable library of screens, flows, and marketing sections captured from shipped mobile and web products. This audit used the official, OAuth-authenticated Mobbin MCP (verified live and Pro-tier) to sample real screens against Meridian's actual, specific requirements — session revocation, tenant/organization context switching, role and permission administration, product/inventory management, progressive disclosure, and a handful of adjacent areas — and to translate observed patterns into original Meridian recommendations, never into copied designs.

**Where Mobbin adds value.** It is strongest as *comparative evidence*: seeing how Vercel, GitHub, Dropbox, PlanetScale, Shopify, Deel, and Cloudflare each solve the same narrow problem (session revocation, audit logging, role matrices, stock receiving) surfaces genuine disagreements between mature products that a single competitor analysis would miss, and it makes Meridian's own governance rules easier to stress-test against real precedent rather than hypothetical scenarios. It is also useful for confirming that a rule already in Meridian's UX governance (e.g., "no more than two persistent navigation levels," `PDA-UX-020` rule 7) is not an academic constraint — real B2B SaaS products routinely violate it, and the ones that don't (Linear, Vercel) are demonstrably easier to navigate.

**Where it does not add value.** Mobbin returns screenshots and short metadata (app name, platform, a permanent URL) — no text, no OCR, no interaction traces, no accessibility information, no state-machine detail, and (per its own tool documentation) no reliable per-app filtering tool beyond naming the app in a natural-language query. It cannot verify a pattern's accessibility, security, or actual task-completion time; it can only show what a product's designers chose to ship. Several of this audit's sample queries returned screen-builder tool UIs (Retool, Glide) rather than real end-user products when the query concept ("wizard," "admin product table") was too generic — this is recorded as a limitation, not papered over with weak evidence.

**Strongest observed principles**, consistent across the sample: (1) mature B2B products separate the *authorization decision* ("can this role do X") from the *entitlement decision* ("does this plan/tenant include X") in their UI language, exactly matching Meridian's existing rule; (2) audit logs that show before/after values inline (Dropbox, PlanetScale) are read faster than logs that only name the action; (3) CSV/data import review is universally a three- or four-step wizard with a persistent step tracker and an explicit "N valid / M needs review" count — never a single blocking form; (4) session-management pages that show device, location/IP, and last-activity together are trusted more than ones that only show a device name.

**Major mobile vs. web differences observed**: mobile settings pages flatten Meridian's proposed two-level admin structure into a single scrollable list grouped by section headers (Wise, Acorns, Expedia) rather than nested drill-downs — this is evidence *for* Meridian's existing "mobile navigation preserves the task, not the desktop layout" rule (`PDA-UX-020` rule 14), not a new discovery. Mobile bottom-tab navigation caps at 4–5 destinations everywhere sampled (ClickUp, Deel, Notion) — none used a hamburger-only pattern for primary navigation.

**Major risks**: two anti-patterns kept recurring and are worth explicit rejection (§13) — paywall modals that blur/obscure the actual content behind them (Uxcel), and product-catalog patterns that are genuinely consumer storefronts (Klarna, Nike, HODINKEE) rather than back-office admin tools, which this audit initially mis-sampled and had to explicitly re-query and flag (§4, §9) rather than let stand as false enterprise-catalog evidence.

**Recommended bounded prototype areas** (detailed in §17): the WS1 application shell / context switcher, a session-management view, permission-vs-entitlement denial states, the WS2 product list/filter/pagination shell, and one mobile/offline workflow.

**Final recommendation** (§18): Mobbin should be used as a governed pattern-research source — not a design authority, not a component library, and not evidence of accessibility or security. See the final scoring in §18.

## 2. Access and Methodology

**MCP used**: the official Mobbin MCP (`com.mobbin/mobbin`, endpoint `https://api.mobbin.com/mcp`, Streamable HTTP transport, OAuth), registered via `claude mcp add mobbin --scope user --transport http https://api.mobbin.com/mcp` and authorized interactively by the account owner in a separate session (this session cannot run OAuth flows).

**Date**: 2026-07-14.

**Access level**: authenticated, Pro-tier confirmed live by a real query returning current production screens (Indeed, Vercel, Wise) rather than stub/cached data, including a Vercel MFA-setup screen reading "2FA is on. Add a passkey for a better multi-factor setup... The best shippers enable both" — directly on-topic for a real Meridian requirement, which is strong evidence the account has genuine search access rather than a degraded/free-tier response.

**Tools actually exposed** (inspected directly from the connected MCP's tool schemas — not assumed from third-party documentation, several pieces of which describe tools this server does not have):

| Tool | Parameters | Returns |
|---|---|---|
| `search_screens` | `query` (natural language, one screen), `platform` (`ios` \| `web`, required), `mode` (`deep` default), `limit` (≤30), `exclude_screen_ids` | Inline images (webp/jpg) + `{id, image_url, mobbin_url, app_name, platform}` per screen |
| `search_flows` | `query` (one user journey), `platform` (required), `limit` (≤10), `page` (≤20) | Inline per-screen preview images + `{id, name, actions, mobbin_url, app_name, platform, screen_count, screens[]}` |
| `search_sections` | `query` (one website section), `limit` (≤30), `page` | Inline images + `{id, image_url, mobbin_url, site_name}` (no platform filter — website sections only) |

No `quick_search`, `get_app_screens`, `get_app_flows`, `get_screen_detail`, or `get_filters` tool exists on this server, contrary to several third-party write-ups describing an unofficial fork. There is no dedicated app-resolution tool; filtering to one app is done by naming it inside the natural-language `query` string. No text/OCR/interaction data is ever returned — every screen had to be visually interpreted by the model performing this audit, not machine-read.

**Search limitations observed in practice**:
- `search_flows` and `search_screens` accept only `ios` or `web` as the platform value — there is no `android` value and no dedicated desktop-vs-web distinction.
- No visible rate-limit signal was returned by any call in this session.
- Generic queries about internal admin-tool concepts (e.g. "multi-step wizard... review and confirm," "admin back-office product table") sometimes returned no-code screen-builder products (Retool, Glide) building *generic* wizard/table components rather than real end-user products using those patterns. These results were excluded from the evidence register (§4) rather than cited as if they were genuine enterprise-workflow precedent; the queries were repeated with the requesting app category named explicitly where that fixed the problem, and left as a recorded gap where it did not (see §4 notes on MOB IDs affected).
- The first product-catalog query ("product catalog list page with filters...") returned exclusively consumer storefronts (HODINKEE, Klarna, Nike). The brief explicitly warns against treating consumer shopping catalogs as enterprise catalog/inventory UX evidence — this audit flags those results as consumer-only in §4/§9 rather than silently reusing them as back-office evidence, and re-queried with "admin back office... not a storefront" phrasing, which materially improved (but did not fully fix) the mix.

**Sample method**: requirement-driven, not exhaustive inventory. For each Meridian need identified from the governing docs (§3, §6), 3–4 real screens or one real flow were retrieved from named, diverse apps (never the same 2–3 apps repeated across queries) and visually reviewed. 20 distinct queries were run across the priority scope; ~50 distinct real apps were sampled (no single app appears in more than 3 of the 40 register rows below). Sections D (POS), G (analytics), H (AI), I (docs/onboarding), and J (marketing) received deliberately lighter coverage than A (WS1 identity/tenancy — already built, so this is validation-plus-polish) and C (WS2 catalog/inventory — genuinely greenfield next work), consistent with the brief's own priority ordering and WS1/WS2's actual implementation state (§3).

**Legal/safety boundary**: this audit operated only through the connected, authenticated MCP, at Mobbin's own permitted internal-business-use scope. No crawling, scraping, bulk download, mirroring, or reverse engineering occurred. No screenshot, video, prototype, Figma asset, or long copied text was written to any file in this repository — every reference below is a short paraphrased observation plus a `mobbin_url` citation, per Mobbin's own `search_screens` tool instruction to always cite that URL when presenting results. No Mobbin credential, token, cookie, or private collection URL was logged, displayed, or committed at any point.

**Confirmation**: no assets, screenshots, or secrets were committed by this audit. The only files this audit adds or edits are Markdown governance documents (this file, and the narrowly-scoped edits proposed in §16).

## 3. Meridian Baseline

Read directly from the repository at commit `1bda97d` (branch `main`), not from documentation claims:

**Routes** (`apps/web/src/app`): `/` (a minimal "Controlled Prototype" health-check landing page, not a product homepage), `/login` (real — sign-in/sign-up toggle, handles `?reason=session-revoked` and `?returnTo=`), `/dashboard` (**dead code** — the route is a bare `redirect("/administration")`; an orphaned `dashboard.tsx` component exists but is not wired to anything), `/administration` and its five real sub-pages (`/users`, `/roles`, `/entitlements`, `/sessions`, `/audit`), all served through a shared layout that performs a server-side session check and renders a tenant/organization/location context bar (`workspace-context.tsx`). **No settings pages exist anywhere in the app.**

**Shared components** (`packages/ui-web/core`): 14 files, all shadcn/Base-UI-style primitives (button, card, dialog, table, sheet, dropdown-menu, etc.). Zero composite/domain components — the admin UI's actual user table, role picker, and audit viewer all live ad hoc inside a single 663-line `administration-pages.tsx`, not as reusable, tested components. **Storybook is not configured anywhere in the repository** — no `.storybook/` directory, no `*.stories.*` file, no `storybook` dependency in any package, despite `PDA-UX-032`'s acceptance checklist requiring Storybook evidence for every Platform Approved component. This is a real, current gap, not a WS1-in-progress item.

**WS1 maturity**: PRs 1–8 are merged (27 total merged PRs, 0 open at time of writing); the server (`apps/server/src/router.ts`) has real, permission-gated oRPC endpoints for session listing/revocation, role assignment, entitlement listing, and tenant-scoped audit listing — this is not stubbed. WS1 PR9 (evidence closeout) has not yet been opened as a PR; it appears to be staged locally elsewhere and this audit does not touch it, per the brief's instruction not to block active WS1 implementation packages.

**Design-system maturity**: `registry/design-tokens.json` is `status: "draft"`, with 6 token categories (space, radius, motion, screen, size, color) totaling roughly 85 leaf tokens, including 5 named motion-duration roles (`instant`/`fast`/`normal`/`slow`/`deliberate`) that `PDA-UX-036` governs explicitly.

**Known gaps this audit's recommendations must account for**: no dashboard, no Storybook, no settings surface, and a component library that is 100% primitives with zero shared composites — meaning any Mobbin-informed recommendation for a composite pattern (context switcher, session list, role editor) is proposing genuinely new Meridian-owned work, not a normalization pass over an existing candidate.

## 5. Navigation Synthesis

**Primary navigation model**: no change recommended. Meridian's existing five-layer model (`PDA-UX-020`) — primary navigation, one nested capability-group level, contextual tabs/local sidebar, record navigation, and search/palette as an accelerator — is already stricter and more precise than every sampled real-world product except Linear and Vercel. The Mobbin sample repeatedly confirmed the failure mode this model exists to prevent (MOB-022, Zoho CRM's three-plus-level sidebar) rather than surfacing a better alternative model.

**Nested-menu rules**: confirmed as-is (`PDA-UX-020` rule 8, one coherent capability group per submenu). No sampled product suggested a better grouping heuristic than "one coherent capability group."

**Contextual navigation**: MOB-033 (Squarespace side panel) is the strongest reusable evidence — a right-side inspector panel that preserves list context is directly applicable to Meridian's audit and product-list rows without introducing a new persistent navigation level.

**Context switching**: MOB-006/MOB-007 (Figma, Linear) confirm Meridian needs two distinct surfaces — a global `ContextSwitcher` (which tenant/org/location am I in) and, later, per-record scope indicators (which org/location does *this record* belong to) — never one overloaded control. The existing `workspace-context.tsx` component already implements the global switcher; no redesign is recommended, only the addition of MOB-006's "show entitlement summary inline" idea as a future enhancement, not a WS1 requirement.

**Mobile transformation**: confirmed as-is (`PDA-UX-020` rule 14; MOB-037). Recommend one numeric clarification (§16): a 4–5 destination cap for mobile bottom-tab navigation, evidenced by MOB-038.

**Search/command-palette role**: MOB-023 (Vapi) is strong positive evidence that a command palette can include an org-switch action *in addition to* — never instead of — a persistent, always-visible switcher, directly confirming `PDA-UX-020` rule 12. No command palette exists in Meridian today; this is recorded as a future (post-WS1) candidate, not a current gap.

**Breadcrumb behavior**: no strong evidence gathered this round — recorded as insufficient evidence, not force-fit from weak samples.

**Recent/favorite behavior**: no strong evidence gathered this round for Meridian's specific administrative context; several sampled consumer apps show recent-items lists, but none in an enterprise-admin context relevant enough to cite as evidence-grade.

## 7. WS1 Recommendations

- **Shell / context switcher**: no redesign; MOB-006/MOB-007 confirm the existing `workspace-context.tsx` global-switcher approach is correct. Consider (non-blocking, future) an entitlement-summary badge inline in the switcher per MOB-006.
- **Sessions**: build `SessionList` (§15) showing device, coarse location (not precise geolocation), last-activity, and a per-row Terminate action, per MOB-003; keep it visually and structurally separate from any future auth-factor list, per MOB-004.
- **Users/invitations**: capture role at invite time in the same control as the email field (MOB-008); if role count grows past ~4, prefer a dedicated modal (MOB-009) over an inline row.
- **Roles**: `RoleAssignmentEditor` must map each permission id to a governed human-readable label + description (MOB-011), and must show an explicit "N members affected" warning before a role's permission set change commits (MOB-010).
- **Entitlements**: `EntitlementUnavailableState` must show the specific capability id and, where limit-based, current usage against the limit (MOB-017); must never blur or partially obscure gated content (MOB-018, rejected).
- **Audit**: `AuditTimeline` renders `changeSummary` diffs inline per row (MOB-012), shows the exact registered event name on hover/expand alongside a human sentence (MOB-013), and supports filter-by-actor/filter-by-outcome with text-labeled (not color-only) badges (MOB-014).
- **Permission denied**: explain which permission is missing, in one sentence, with no dead end (MOB-015/016); do not build a "request access" action in WS1 since no approval-request endpoint exists yet.
- **Entitlement unavailable**: see above (MOB-017/018).
- **Reauthentication**: extend the existing `?reason=` query-param handling in `apps/web/src/app/login` to distinguish inactivity-timeout from explicit user-revoked from admin-suspended, per MOB-019; keep the existing full-page-redirect approach rather than adopting Square's in-context modal (MOB-020) unless a specific unsaved-state-preservation need is identified.

## 8. WS2 Recommendations

- **Product list**: row action menu separates destructive from non-destructive actions with a visual divider (MOB-025); do not reuse consumer-storefront filter *content* (MOB-024, rejected) though the general faceted-filter *mechanism* is common ground.
- **Product detail / variants**: `ProductVariantMatrix` must apply `PDA-UX-019`'s existing row-count pagination/virtualization thresholds, since multi-axis variant generation can produce combinatorial row counts (MOB-026).
- **Filtering**: no new pattern beyond existing `PDA-UX-011`/`PDA-UX-019` guidance identified this round.
- **Pagination**: no new evidence beyond existing `PDA-UX-019` guidance (cursor-based, per the platform's OpenAPI contract) — Mobbin cannot evidence Meridian's own API-contract constraint, since that constraint is internal, not a UX pattern question.
- **Product form**: no strong new evidence this round.
- **Stock by location**: the on-hand/committed/available three-way split (MOB-027) should be the assumed data model for any stock-by-location UI.
- **Adjustments/receiving**: the strongest evidence in this entire audit — accept/reject quantity per line against an ordered baseline, a persistent completion-fraction indicator, and (from Fresha) a cost-reconciliation summary alongside quantities (MOB-028, MOB-029).
- **Transfers**: no dedicated transfer-specific evidence gathered distinct from the receiving pattern above; recommend reusing the receiving flow's accept/reject-per-line shape for transfers pending a dedicated research pass if WS2 planning reveals transfer-specific needs the receiving pattern doesn't cover.
- **Counts**: no strong evidence gathered this round — recorded as insufficient evidence rather than force-fit.
- **Reconciliation**: no strong evidence gathered this round.
- **Activity**: reuse the `AuditTimeline` recommendations from §7 — the same before/after-value and structured-event-name patterns apply to a product/inventory activity feed.
- **Import**: three-part synthesis — explicit valid/excluded row counts before commit (MOB-030), inline per-cell correction with the *same* validation as the bulk path if built (MOB-031), and a persistent step-tracker for multi-step imports (MOB-032).

## 9. POS Findings

Coverage was deliberately light (POS/Prototype 3 has not started; `PDA-UX-010`'s tender-scope work is separately governed and already detailed). The one targeted POS query (cash-register close/variance review) returned consumer banking/fintech flows (Cash App, KakaoBank, Kraken withdrawal flows) rather than genuine cashier register-close workflows — **recorded as insufficient evidence, not force-fit as POS precedent.** This is an explicit example of the search-quality limitation named in §2: Mobbin's index appears to have limited coverage of retail/POS-specific back-office workflows compared to its strong coverage of general SaaS admin and consumer fintech patterns.

- **Reusable pattern insight**: none confirmed this round with sufficient evidence quality.
- **Unsuitable consumer-checkout patterns**: the sampled consumer banking withdrawal flows (Kraken) use a numeric-keypad-first, single-value-entry pattern appropriate for a single withdrawal amount — this does not transfer to POS's multi-line, multi-tender cash-count-and-variance need, which is structurally different (many denominations, not one amount).

Recommend a dedicated, better-scoped Mobbin query (e.g. naming specific POS/retail products if any exist in Mobbin's index) before Prototype 3 planning, rather than relying on this audit's thin result.

## 10. Mobile Findings

- **Patterns that do not translate to desktop**: bottom-tab primary navigation itself is mobile-only; Meridian's desktop admin already correctly uses a persistent sidebar instead, and no sampled evidence suggested changing that.
- **Patterns requiring Expo/native implementation**: out of scope — Meridian's native app (`apps/native`) has zero auth/admin code today per prior session findings, and this audit's mobile sampling was `ios` web-app screenshots via Mobbin, not native-app-specific research. A dedicated native-focused Mobbin pass is recommended before any native WS1+ work starts, not substituted by this audit's `ios`-platform web screenshots.
- **Accessibility and offline risks**: Mobbin provides no accessibility or offline-behavior data at all (§2) — every offline/degraded-state requirement in `PDA-UX-010` remains entirely ungrounded in this audit and must be verified through Meridian's own testing, never assumed from a screenshot.

## 17. Bounded Prototype Batch

1. **WS1 application shell / context switcher enhancement.** Learning objective: confirm an inline entitlement-summary badge (MOB-006) improves context clarity without implying a marketing "plan name" that doesn't match Meridian's capability-based entitlement model. Implementation source: enhancement to existing `workspace-context.tsx`. Risks: could imply a simple tiered-plan model that doesn't exist. Acceptance evidence: usability check that testers correctly describe their current tenant's entitlement summary without confusing it for a billing plan name. Removal criteria: if testers consistently misinterpret the badge as a plan name, revert to the current switcher with no badge.
2. **Session-management view (`SessionList`).** Learning objective: validate that showing device + coarse location + last-active + revoke (MOB-003) gives users enough evidence to recognize a suspicious session without over-precise geolocation. Implementation source: new custom composite, backed by the already-real WS1 PR7 session-revocation API. Risks: coarse-location data may not be available from the current infrastructure; must degrade gracefully to device-only if so. Acceptance evidence: a controlled test where a tester is shown a seeded second "session" and can correctly identify and revoke it using only the UI's shown fields. Removal criteria: if location data cannot be sourced reliably, ship device+last-active only rather than fabricating location.
3. **Permission-vs-entitlement denial states.** Learning objective: confirm `PermissionDeniedState` and `EntitlementUnavailableState` are visually and semantically distinguishable to end users (a WS1 exit-gate requirement already, per the merged WS1 plan). Implementation source: new custom composites (§15). Risks: if the two states look too similar, users will file support tickets for the wrong root cause. Acceptance evidence: a card-sort/labeling test where testers correctly distinguish the two states from screenshots alone. Removal criteria: none — this is a WS1 exit-gate requirement, not optional.
4. **WS2 product list/filter/pagination shell.** Learning objective: validate the row-action-menu destructive/non-destructive separation (MOB-025) and cursor-based "load more" pattern against Meridian's actual OpenAPI cursor-pagination contract. Implementation source: new custom composites (`ProductListToolbar`, `CursorPagination`), built once WS2 begins. Risks: none identified beyond ordinary WS2 scope risk. Acceptance evidence: a tester can locate and safely execute a destructive bulk action without accidentally triggering it via a misclick, in a moderated usability session. Removal criteria: none — this is foundational WS2 UI, not a throwaway prototype.
5. **One mobile/offline workflow.** Learning objective: since Mobbin provided zero offline/accessibility evidence (§2, §10), this prototype exists specifically to generate Meridian's own first-party evidence rather than relying on any external source. Implementation source: whichever WS1/WS2 workflow is selected should be an already-real one (e.g. the existing `/administration/sessions` page) tested under a simulated offline/degraded network condition. Risks: none beyond ordinary testing risk. Acceptance evidence: the workflow correctly shows Meridian's existing offline/degraded canonical states (already required by `PDA-UX-031`) rather than a blank or broken screen. Removal criteria: none — this generates required first-party evidence Mobbin cannot supply, not a discretionary prototype.

## 18. Final Recommendation

**Mobbin should be used as a governed pattern-research source.**

Rationale: across 20 queries and roughly 50 sampled apps, Mobbin repeatedly did two useful things — it gave concrete, diverse real-world precedent that either *confirmed* an existing Meridian rule with evidence (the majority of this audit's findings: MOB-010, MOB-014, MOB-017, MOB-022, MOB-023, MOB-035, MOB-037, MOB-038) or surfaced a genuinely new, narrow, evidence-backed implementation detail worth adopting (MOB-003/004 session-list shape, MOB-011 permission-label mapping, MOB-012/013 audit-diff rendering, MOB-025 row-action separation, MOB-026/027/028/029 the inventory/receiving model). It never once produced evidence strong enough to justify overriding or loosening an existing Meridian rule, and one sample (MOB-018) produced a clear anti-pattern worth explicitly rejecting.

It falls meaningfully short of "required research step" status because: (1) its search surface is uneven — POS/retail back-office coverage and generic wizard-concept queries both returned weak or off-target results this round (§9, MOB-040), meaning it cannot be relied on as the *sole* research input for every workflow; (2) it supplies zero accessibility, security, or offline-behavior evidence, which are load-bearing requirements for nearly every Meridian component (§10); and (3) it is, by its own terms of use and by this audit's own governance proposal (§16), explicitly non-authoritative — a screenshot is never proof that a pattern is correct, only that it was shipped somewhere.

"Occasional inspiration" undersells it: this audit's register (§4) shows it can be pointed at a specific, named Meridian requirement and reliably return diverse, comparable real-world evidence, which is more structured and repeatable than ad hoc inspiration browsing. "Mobbin should not be used" is not supported by the evidence gathered — it produced zero false-positive Platform-Approved-worthy patterns and one genuinely load-bearing methodology correction (catching its own consumer-catalog mis-sample, §4/§9) when used carefully.
