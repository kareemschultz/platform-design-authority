## 1. Executive Verdict

**The remediation engine works.** 26 of 31 second-audit findings are genuinely Closed with verifiable content, 3 are Partially closed exactly as (or nearly as) the disposition declares, 0 regressed. All four new ADRs (0013 stored value, 0014 PII isolation, 0015 direct merchant contracts, 0016 namespaces and events) propagated cleanly through the repository — the "asserted-but-unpropagated" failure mode that defined the second audit is substantially cured. Governance automation is now real: repo-wide event linting, internal-link validation, related-ADR integrity, registry freshness with a curated first-slice overlay, and strictly read-only CI.

**The failure mode has shifted again.** The first audit found missing architecture; the second found unpropagated decisions; this audit finds **declared-but-uninstantiated contracts**. The first-slice package is coherent as prose but fails its own machine checks: half its declared events (including `commerce.sale.completed.v1`) exist in no registry, the API contracts cover roughly half the flows, the permission catalog covers neither the APIs nor whole capability groups, and `registry/first-slice.json` encodes a materially larger scope (104 capabilities including `finance.general-ledger`) than the manifest promises. Meanwhile the entire quality-gate system — experience budgets, SLOs, error budgets, RPO/RTO — contains **zero numeric values**, making three separate enforcement mechanisms ceremonial.

**Readiness:** Ready for one constrained vertical-slice implementation **after the named blockers in §19 "Now"** — with Technical Prototypes 1–3 (identity/tenant context, catalog/inventory ledger, POS cash) able to start immediately. Not ready for pilot: pilot requires the founder decisions (FDR-001/002/003), authoritative Guyana legal verification, provider certification, and operational evidence that documentation cannot supply — and the repository, to its credit, says so itself.

Strongest aspects: ADR discipline and propagation; commercial money-context separation (8 contexts verified clean); the stored-value specification; backup/restore with deletion-journal watermark; the first-slice AI boundary; honest completeness matrix; zero invented business facts (legal entity, currency, merchant status all correctly held open).

Most serious risks: the numeric-vacuum in quality gates; first-slice registry/contract/permission incompleteness; the marketplace existing outside the repo's own capability governance; zero customer evidence behind a "provisionally adopted" beachhead; Guyana regulatory unknowns (Bank of Guyana licensing guidance unverifiable, Data Protection Act not yet in force, September 2025 FX controls).

## 4–5. Findings by Severity (new findings, TA numbering; contradictions integrated)

### Blocker

**TA-001 — The first-slice event contract evades the event registry; core domain event namespaces are empty.**
`02-Architecture/FIRST_SLICE_API_AND_EVENT_CONTRACTS.md` §Initial Event Families declares 14 events, but the registry generator extracts only from headings named Events / Event Integration / Required Events / Representative Events (`scripts/generate_registries.py` CANONICAL_EVENT_HEADINGS). Seven of the fourteen exist nowhere else and are absent from `registry/events.json`: `catalog.product.created.v1`, `inventory.stock.adjusted.v1`, `inventory.stock-count.posted.v1`, `commerce.sale.completed.v1`, `commerce.return.completed.v1`, `commerce.register.opened.v1`, `commerce.register.closed.v1`. Registry namespace counts: commerce 7 (all stored-value), **inventory 0, catalog 0, finance 0, fiscalization 0, developer 0**. The single most important fact in a POS platform — sale completed — has no canonical definition, no owner resolution, no duplicate protection, and would be invisible to any SDK generation, directly against ADR-0016 §Required Controls. Webhooks and Fiscalization engine docs likewise define no events despite prose describing delivery-failure and submission lifecycles (the SA-008 rehoming deleted `platform.webhook.delivery-failed.v1` and registered nothing in its place).
**Fix:** define these events under canonical `## Events` headings in the owning domain docs (Commerce, Inventory, Catalog, plus Webhooks and Fiscalization engine), make the first-slice contract reference them, and add a CI rule failing any strict-shape event bullet outside canonical headings whose name is not already registered.

### Critical

**TA-002 — `registry/first-slice.json` encodes a bigger slice than the manifest approves.** The manifest scopes Finance to a bounded accountant handoff and "does not promise a complete production general ledger"; the registry includes `finance.general-ledger` and `finance.financial-reporting` among 8 unbounded finance capabilities. The manifest defers ~10 items; the registry records 5 (production fiscal submission — a legal deferral — is not machine-recorded, while `engine.fiscalization` and 4 `fiscalization.*` capabilities sit in the included list unmarked). There is no depth/maturity attribute, so "seam" vs "full capability" is unverifiable and change control cannot detect the drift. Judgment on the 104-capability set: roughly 35–40 are genuinely required for the stated outcome; the remainder are platform ambitions riding along under prose qualifiers the machine artifact does not carry. **Fix:** add a per-capability `depth: seam|prototype|full` attribute (the `first-slice.json` curated-overlay pattern already proves the mechanism), mirror the full deferral list, and hold a founder re-scope review (interacts with FDR-004).

**TA-003 — The API contracts cover roughly half the flows.** `FIRST_SLICE_API_AND_EVENT_CONTRACTS.md` has no endpoint families for: customers/parties (Flow 1 needs party context; `crm.customer.*`/`party.record.*` permissions exist), cash movements/safe drops/deposits (Flow 6; permissions exist), refund execution on returns beyond payment-intent refund, accountant-handoff export (acceptance scenario 10), receipt reissue/void, stored-value redeem/adjust/suspend/reconcile, user/role/entitlement administration, webhook subscriptions, and customer/opening-stock imports. Its own quality gate — "OpenAPI validation" — has no OpenAPI artifact to validate. Acceptance scenarios 6, 9, and 10 have no contract surface.

**TA-004 — The permission catalog covers neither the APIs nor the capability set.** Missing: payment-intent create; offline-lease issue; sync-batch submit; stored-value reserve; privacy-request create/intake (UX acceptance requires staff intake); every finance, fiscalization, and webhook permission; branding and tax/jurisdiction-profile configuration (both in the UX Admin workspace). Reverse direction is clean: all 62 existing permission ids use registered prefixes and 3-segment form. No permission↔endpoint mapping exists anywhere.

**TA-005 — The quality-gate system has zero numeric values.** `20-Strategy/PLATFORM_EXPERIENCE_INDEX.md` requires per-workflow budgets "before implementation" — none exist. `16-Testing` release gates require "accessibility and performance budgets met" — none exist. `15-Operations/SLO_AND_OPERATIONAL_READINESS.md` defines the record format with zero targets, so error budgets cannot function. `12-Deployment/BACKUP_RESTORE_AND_DISASTER_RECOVERY.md` mandates every service declare RPO/RTO — no service has declared any. `CAPACITY_COST_AND_MULTI_REGION_STRATEGY.md` has no pilot workload model. A repo-wide grep for p95/p99/ms/error-rate targets in these sections returns nothing. Three separate enforcement mechanisms are currently ceremony. **Fix:** publish one provisional first-slice numbers table (POS sale-completion latency, search latency, offline-sync age, PITR RPO, restore RTO, pilot capacity envelope, per-workflow time/click/training budgets), marked pre-measurement.

### High

**TA-006 — Entity model missing load-bearing entities.** No Stock Transfer lifecycle (API and permissions exist), no Deposit lifecycle (Flow 6 depends on it), no Exchange (capability and manifest promise it), no webhook subscription entity, and the tax snapshot Flow 1 validates is explicitly deferred by the entity model's Open Schema Decisions — flows depend on a structure the model declines to define even provisionally.

**TA-007 — The jurisdiction profile defers every tax/receipt value the flows need.** VAT "commonly operated as 14%" unverified; receipt mandatory fields, rounding, sequential numbering all "verify before production" with no provisional prototype values fixed. Consequence: the receipt renderer and tax snapshot have no concrete inputs even at prototype fidelity. **Fix:** publish a "prototype tax pack v0" with fixed, clearly-non-statutory provisional values so implementation proceeds while verification runs.

**TA-008 — Jurisdiction corrections from live research (2026-07-11).** (a) Guyana's **Data Protection Act 2023 is not in force** — no commencement order found; Commissioner's office only being established as of May 2026 (source limitation on exact status). Design for compliance; do not claim operative law. (b) **September 2025 FX controls** (invoice verification before FX approval, reported rate caps ~GYD 216.25–216.50/USD, USD cash limits) materially affect USD settlement assumptions (US State Dept 2025 Investment Climate Statement). (c) **No Guyana e-invoicing/fiscalization mandate found** (EY global tracker June 2026 does not flag Guyana; beware gra.gov.gh Ghana sources) — the fiscalization engine's justification must read "export-market/future-proofing (EU ViDA: Directive 2025/516 in force 2025-04-14, mandatory intra-EU structured e-invoicing July 2030)", not local mandate. (d) GRA verified: VAT 14%, registration threshold GYD 15,000,000/12mo, monthly returns due within 15 working days, 6-year retention, tax-invoice field list. (e) NIS verified: 14% (5.6% employee / 8.4% employer), ceiling GYD 280,000/month **but the NIS site still shows the Jan 2020 value — verify before payroll work**. (f) Bank of Guyana: National Payments System Act 2018 and PSP Guideline No. 1 (2023) verified; **MMG is a licensed PSP (licence 1/2024**, alongside Caripay and Kanoo); no published guidance on aggregation/facilitation licensing — legal advice required, exactly as ADR-0015 records.

**TA-009 — Vendor/stack verification corrections.** (a) **Better Auth now ships a first-party SCIM plugin** (`@better-auth/scim`) — `19-Appendices/BETTER_AUTH_VERIFICATION-2026-07-10.md` predates it and needs a dated update; SAML 2.0 is in the **free** SSO plugin; the paid "Infrastructure" tier ($20/mo Pro; $50/mo per extra self-service SSO connection) covers self-service dashboards, not protocol capability; current release v1.6.23 (2026-06-29). (b) **TanStack Start is still RC, not GA** — any GA assumption is premature. (c) **Expo UI is stable as of SDK 56** — any alpha/beta caveat is stale. (d) Node 24 is Active LTS; PostgreSQL 18 current; Redis 8.x is tri-licensed AGPL/SSPL/RSAL with Valkey 9.1 (BSD) the unencumbered alternative; NATS remained Apache-2.0 under CNCF after the 2025 Synadia settlement; Temporal server 1.31.x; Better-T-Stack actively maintained (v3.36.3, 2026-07-08). (e) MMG e-commerce API verified real but thin: hosted redirect checkout with callback URL and a Merchant Initiated Payment API; **recurring/auto-debit, server-to-server webhooks, API refunds, sandbox, and settlement SLAs remain publicly undocumented** (consumer refunds are a paper process) — the rails doc's Known-Unknowns list is confirmed correct. (f) Stripe re-verified: Guyana not supported; Atlas/US-entity is the Stripe path, consistent with FDR-002's framing.

**TA-010 — Better Auth identity doc contradicts the Party ownership split.** `01-Platform/BETTER_AUTH_IDENTITY_ARCHITECTURE.md` §Architectural Position says the identity/Party/authorization layers own "Workforce, customer, supplier, and external-party role records" — ADR-0007 and the Party model assign role records to domains. Rewrite as "links to domain-owned role records."

**TA-011 — Workforce domain never aligned to ADR-0007.** The only Part-1 domain doc with no Party reference: "Worker" is declared authoritative with no statement that it is a role projection on a canonical Party; no `related_adrs: [ADR-0007]`; still v0.1.0 while CRM/Procurement were bumped in the alignment pass. This is the duplicate-person-record seam ADR-0007 exists to close.

**TA-012 — The marketplace is unregistered in the repo's own capability governance.** Two spec docs and an index, but no `marketplace` namespace in `domains.json`, zero marketplace capabilities in the map or registry — the exact SA-009 defect class, one batch later. Register a `marketplace.*` family (publishing, review, installation, licensing, settlement) or file under `commercial.*`/`developer.*` with a boundary sentence.

**TA-013 — Project skills: wrong portfolio, and the tool-restriction mechanism doesn't do what the skills assume.** (a) Audit 1's accepted recommendation (spec-author, adr-author, consistency-audit, capability-registrar) produced zero of those; the six shipped skills are all frontend-phase, and `frontend-architecture` declares path globs (`apps/**`, `packages/ui-*/**`) that don't exist in the repo. The repository's daily work — spec authoring, registry regeneration, consistency auditing — has no skill support. (b) Verified against official docs: **`allowed-tools` grants pre-approval; it does not restrict** — "every tool remains callable." Five skills use `allowed-tools` with restrictive intent; actual restriction requires `disallowed-tools` or permission deny rules. Also `context: fork`/`agent:` may be ignored on model-invoked paths (open issue anthropics/claude-code#17283). (c) `frontend-architecture` is additionally the only skill with no tool fields and no fork, contradicting the repo's own PROJECT_AGENT_SKILLS rules 6–7, and uses an undocumented `paths:` key. Positive: no authority escalation and no proprietary prompt copying in any skill (verified clean).

**TA-014 — AI platform: guardrails are real, mechanisms are not.** (a) Provider portability is asserted (Principle 7, registry `fallback` field, GA-gate "provider exit plan") with no defined exit-plan contents, no requirement that a fallback pass the same evaluation suite, no rehearsal. (b) Loop control is a prohibition without a mechanism: no delegation depth ceiling, no cycle detection, no defined behavior at limit; "agents cannot delegate more authority than they possess" has no enforcement design. (c) `AI_SDK_MULTI_AGENT_AND_MUTATING_AGENT_CONTROLS.md` never states whether it *is* the "separate future specification" the first-slice boundary requires for mutating agents — an implementer cannot tell if the precondition is satisfied. Positive verified: tool authority through ordinary permissions is consistent across 5+ docs; ADR-0014 erasure linkage covers prompts/embeddings/memory/traces in all five relevant documents; retrieval authorization is contradiction-free across Search, Memory, AI boundary, and threat model.

**TA-015 — Marketplace AI packs bypass the AI governance pipeline.** `EXTENSION_PLUGIN_AND_SANDBOX_ARCHITECTURE.md` lists "AI skill or tool pack" as an extension type, but `06-AI/MODEL_PROMPT_TOOL_AND_AGENT_REGISTRIES.md` governs only platform registries — no statement that publisher-supplied prompts/tools/agents enter the same registries, pass the same evaluation gates, or suspend with the publisher.

**TA-016 — Zero customer evidence behind a present-tense beachhead.** `COMPANY_AND_PLATFORM_STRATEGY.md` asserts beachhead advantages as established fact; `MARKET_SEGMENTATION_AND_BEACHHEAD_EVIDENCE.md` contains an evidence *schema* with no evidence instances; FDR-004 provisionally adopts first-slice scope with no recorded customer input. The docs mostly admit this (the README lists interviews as remaining evidence) — but the strategy tree currently contains frameworks for evidence and not one datum. Interview count should be a tracked gate metric before prototype investment hardens into pilot commitments.
